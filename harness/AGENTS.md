## Operating model

This repository uses a spec-driven development pipeline. Every non-trivial change starts from a written spec in `harness/specs/<name>.md` and flows through five agent stages, each gated by the next:

1. `spec-reader` reads the spec and produces a structured implementation plan.
2. `architect-review` validates the plan for layering, security, and convention compliance.
3. `implementer` builds the feature task by task against the approved plan.
4. `test-writer` adds unit, integration, and property-based tests against the implemented code.
5. `code-reviewer` performs the final sign-off review.

Pipeline state is persisted in `harness/queue/agent-queue.json`. Hooks automatically advance the queue between stages and surface the next action after every agent turn.

To start a new feature, run the `/implement <spec-name>` command. To check progress, run `/pipeline-status`.

## Conventions

These conventions apply to every change made by any agent.

### Code

- Validate every input at the system boundary; treat everything from outside as untrusted.
- Prefer Result-style error handling over thrown exceptions for predictable, recoverable failures.
- Prefer soft deletes (a `deletedAt` column or equivalent) over hard deletes whenever data is auditable.
- Group multi-record writes in a single transaction; partial writes are never acceptable.
- Avoid magic numbers; extract them into named constants.
- Keep functions small and single-purpose; if a function name contains "and", split it.
- No `any` (or its language equivalent); narrow `unknown` instead.

### Naming

- Files: follow the language's idiomatic case (`camelCase.ts`, `snake_case.py`, etc.).
- Booleans read like questions: `isValid`, `hasAccess`, `canExecute`.
- Functions are verbs: `calculateTax`, `validateInput`.
- Variables are nouns: `userConfig`, `pendingCount`.

### Security

- Never hardcode secrets; read them from environment variables or a secret store.
- Always scope queries to the calling user's tenant; never trust client-supplied identifiers for authorization.
- Sanitize logs; redact PII, tokens, and credentials before they reach a log sink.

### Testing

- Every new public function or endpoint gets at least one success and one failure test.
- Use the Arrange / Act / Assert structure in test bodies.
- Tests must be isolated; one test's state must never bleed into another.

## Documentation

Project documentation lives in `README.md` at the repository root. When a change alters setup, configuration, or runtime behavior, the README must be updated in the same change set. Stale documentation is worse than no documentation.

## What lives where

- `harness/` — the tool-agnostic source of truth for agents, commands, hooks, skills, specs, plans, and queue state. Edit here, run `harness sync`.
- `harness/specs/` — input specifications (one file per feature).
- `harness/plans/` — outputs from `spec-reader`, `architect-review`, and `code-reviewer`.
- `harness/skills/` — reusable knowledge packs that agents auto-load when relevant.
- `.claude/`, `.cursor/`, `.github/`, root `AGENTS.md` — generated by `harness sync`. Do not edit by hand.
